# Apply the initial Email Extractor schema to VPS staging and exercise the integration test

## Goal

Deploy the initial domain schema (commit `7f43073`, Alembic revision `78f509b95848`) to VPS staging: pull the new code on the VPS, run `alembic upgrade head` inside the backend container, restart the backend so it loads the new endpoints, then curl-smoke `POST /scans` and `GET /scans/{id}` against `http://76.13.22.110:8000` to verify the round-trip works end-to-end. Finally, run the pytest suite with the `integration` marker enabled against the live VPS DB (via SSH tunnel or directly inside the VPS backend container) to flip the `@pytest.mark.integration` test from "skipped by default" to "exercised at least once." No code changes in this prompt — pure deploy + verification.

## Context

- Current VPS state: main branch at commit `c3f7efa` (pre-domain-models). Backend container is running the old code with no `/api/v1/email-extractor/*` routes and no domain tables in Postgres.
- What lands with this prompt: the three domain tables (`extraction_run`, `discovered_email`, `email_verification`) materialize in the live `email_extractor` DB on the VPS, and the backend image picks up the new endpoints + stub aggregator.
- Why the deploy is its own prompt: prior prompt (`2026-04-19-1123-initial-domain-models.md`) deliberately deferred the VPS apply because it was written assuming a local Docker stack we don't have, and shipping schema-plus-code in the same breath as authoring them felt risky. We now have a reviewed, tested, committed migration — safe to apply.
- The migration was autogenerated against a throwaway DB (`email_extractor_autogen`) on the same Postgres instance we're deploying to, via SSH tunnel. Engine version, collation, extensions — all identical. Reversibility (`up → down → up`) was verified on the throwaway. Applying to `email_extractor` is mechanically the same operation, just against a different DB name.
- CC CLI's Outcome flagged one deviation: parent uses `Mapped[int]` + `autoincrement=True` PKs, not UUIDs (saved to `.auto-memory/reference_parent_pk_convention.md`). Migration shape reflects the corrected choice.

## Constraints

- **Zero code changes in this prompt.** No file edits. Pure ops.
- **No new commits locally.** Only the VPS-side operations. If anything here requires a code change (e.g. a missed config), stop and surface — we'll land that as its own prompt.
- **VPS hygiene unchanged.** Rebuild only `backend` — don't touch `postgres` or `frontend`. No `docker compose down`, no `--volumes`.
- **No `docker compose down postgres`.** The DB must stay up through the migration. Taking it down erases the migration history across a restart (volume persists, but this is a belt-and-braces rule).
- **Back up the VPS DB before the migration.** It's one table with likely zero rows, but the habit matters — pg_dump the DB to a timestamped file in `~/apps/email-extractor/backups/` first. If the migration goes wrong, rollback is a `pg_restore`.
- **If `alembic upgrade head` fails**, do not retry blindly. Capture the error, run `alembic downgrade base` to clean up any partial state, and surface. The migration either applies cleanly or doesn't apply at all.
- **Integration test gating:** the new `@pytest.mark.integration` test in `backend/app/tests/api/test_email_extractor_scans.py` must be run with `pytest -m integration` against the live VPS backend. If pytest isn't runnable from inside the backend container (missing dev deps), run it locally against the VPS API URL via an env-var override — document whichever path CC CLI picks.
- **Respect co-tenants.** Other admins are still on the box. No firewall changes, no sshd edits, no touching ports 80/443/3000/3001/3002/3003/3306.
- **Don't clobber the VPS `.env`.** Nothing in this prompt should touch it.
- **Use the `deploy` user, not `root`.** `ssh deploy@76.13.22.110`.
